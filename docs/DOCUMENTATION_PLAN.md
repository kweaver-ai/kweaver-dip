# 文档设计与规划（面向开源社区）

本文档定义 KWeaver DIP 仓库的文档体系，目标是让新用户在 5-10 分钟内了解项目定位、模块边界和参与方式。

## 设计目标

- **易理解**：首次访问即可看懂“项目是什么、解决什么问题”。
- **易定位**：不同角色能快速找到对应入口文档。
- **易维护**：文档结构清晰，新增模块时有明确更新规则。
- **易协作**：中英文文档并行，兼顾国内外开源用户。

## 信息架构

- 根目录：
  - `README.zh.md`：中文主入口（目标、模块、阅读路径）。
  - `README.md`：英文主入口（对外协作与国际社区）。
- `docs/`：
  - `README.md`：文档导航页。
  - `PROJECT_STRUCTURE.md`：目录职责说明。
  - `DOCUMENTATION_PLAN.md`：文档体系规划（本文件）。
- 其他目录：
  - `release-notes/`：版本变更历史。
  - `rules/`：贡献与协作规范。

## 文档分层策略

- **L1：项目入口层**
  - 目标：回答“这是什么项目”。
  - 文件：`README.zh.md`、`README.md`。
- **L2：结构认知层**
  - 目标：回答“代码在哪里、模块做什么”。
  - 文件：`docs/PROJECT_STRUCTURE.md`。
- **L3：角色操作层**
  - 目标：回答“我下一步该看什么、怎么参与”。
  - 文件：`docs/README.md`、模块级 `README.md`。
- **L4：治理追溯层**
  - 目标：回答“规则是什么、版本变化是什么”。
  - 文件：`rules/`、`release-notes/`。

## 目录文档落地建议

建议为下列核心目录补充模块级 `README.md`：

- `web/README.md`：前端应用边界、技术栈、启动方式。
- `dsg/README.md`：语义治理能力、数据模型、规则定义入口。
- `dip-studio/README.md`：数字员工开发管理流程与核心能力。
- `aistore/README.md`：应用表现层定位、展示模块与发布流程。
- `chat-data/README.md`：数据分析员数字员工能力与交互流程。
- `skills/README.md`：对外技能包目录、版本规范、发布说明。

## 内容模板建议

每个模块级 `README.md` 建议固定包含：

1. 模块定位（一句话说明）
2. 主要能力（3-5 条）
3. 目录说明（关键子目录用途）
4. 本地开发（依赖、启动、构建）
5. 对外接口（如有）
6. 常见问题（FAQ）

## 维护机制

- 新增模块：必须新增模块级 `README.md`。
- 目录职责变化：同步更新 `docs/PROJECT_STRUCTURE.md`。
- 影响用户使用的变更：同步更新根目录 `README`。
- 版本发布：在 `release-notes/` 增加版本说明。

## 里程碑建议

- **M1（当前）**：完成中英文入口页 + 文档导航 + 目录职责说明。
- **M2**：补齐六个核心目录的模块级 `README.md`。
- **M3**：增加部署与开发流程文档，形成可复现上手路径。
- **M4**：完善贡献、FAQ、变更记录，形成稳定开源协作闭环。
